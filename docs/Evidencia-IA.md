## Problem Statement

Build a REST API for MIBICI, Guadalajara's bike-sharing system. The system must support spatial search for nearest stations with real-time availability, handle 10,000 stations and 1,000,000 MAU. This is a take-home technical challenge where the primary evaluation criteria are: spatial indexing (no O(n) search), concurrency safety (race condition testing), stress testing with documentation, and a GitHub Actions CI pipeline runnable with `act`.

## Premises

1. **PostGIS is the spatial backend.** GiST indexes with `<->` KNN operator give O(log n) nearest neighbor, ~1800x faster than sequential scan.
2. **Monolith for Phase 1.** Single FastAPI app, single PostgreSQL/PostGIS database. Microservices are over-engineering for a take-home evaluated with `act`.
3. **FastAPI over Flask.** Async-native, auto OpenAPI docs, Pydantic validation. Strictly better for this use case.
4. **Real MIBICI data.** Seed from the public CSV (~300 real stations). Generate 9,700 synthetic stations as random points within the Guadalajara metro bounding box (20.55-20.75 lat, -103.45 to -103.25 lon) with random `total_slots` 10-30 and `available_bikes` 0 to total_slots, for benchmarking at 10,000 stations.
5. **Advisory locks for concurrency.** `pg_advisory_xact_lock` scoped to transaction. More demonstrable in tests than optimistic locking and shows deeper PostgreSQL knowledge.
6. **Concurrency tests are the primary differentiator.** Race condition proofs and stress test documentation are what separate this from the pack.

## Database Schema

```sql
CREATE TABLE stations (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    location VARCHAR(255),
    geom GEOMETRY(Point, 4326) NOT NULL,
    total_slots INT NOT NULL,
    available_bikes INT NOT NULL DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_stations_geom ON stations USING GIST (geom);

-- Reservations table: tracks individual reserve/return events
-- The spec allows "from a counter to a complex system." We use both:
-- available_bikes counter for fast reads, reservations table for audit trail.
CREATE TABLE reservations (
    id SERIAL PRIMARY KEY,
    station_id INT NOT NULL REFERENCES stations(id),
    reserved_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    returned_at TIMESTAMPTZ,
    status VARCHAR(20) NOT NULL DEFAULT 'active'
);
```

After seeding: `CLUSTER stations USING idx_stations_geom;` then `ANALYZE stations;` for physical data ordering (~90% spatial query speedup per Crunchy Data benchmarks).

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/stations/nearest` | KNN spatial search. Params: lon, lat, k (default 5), radius_m (optional, caps max distance via `ST_DWithin`). When both k and radius_m are set, returns up to k stations within radius_m meters. If fewer than k stations exist within radius, returns only those found. |
| GET | `/stations/{id}` | Get station details + availability |
| POST | `/stations` | Create station |
| DELETE | `/stations/{id}` | Remove station |
| POST | `/stations/{id}/reserve` | Reserve a bike (advisory lock) |
| POST | `/stations/{id}/return` | Return a bike |
| GET | `/health` | Health check for ALB |

## Concurrency Strategy

```python
async def reserve_bike(station_id: int, db: AsyncSession):
    # Transaction-scoped advisory lock: serializes reserves per station
    await db.execute(text("SELECT pg_advisory_xact_lock(:id)"), {"id": station_id})
    station = await db.get(Station, station_id)
    if not station or station.available_bikes <= 0:
        raise HTTPException(409, "No bikes available")
    station.available_bikes -= 1
    await db.commit()
```

```python
async def return_bike(station_id: int, db: AsyncSession):
    await db.execute(text("SELECT pg_advisory_xact_lock(:id)"), {"id": station_id})
    station = await db.get(Station, station_id)
    if not station or station.available_bikes >= station.total_slots:
        raise HTTPException(409, "Station is full")
    station.available_bikes += 1
    # Close the active reservation record
    await db.execute(
        update(Reservation)
        .where(Reservation.station_id == station_id, Reservation.status == "active")
        .values(status="returned", returned_at=func.now())
        .limit(1)
    )
    await db.commit()
```

Why advisory locks over alternatives:
- `SELECT FOR UPDATE`: row-level lock, but can deadlock with multi-row operations
- Optimistic locking (version column): requires retry logic, harder to test
- Advisory lock: explicit, transaction-scoped, no retry needed, zero deadlock risk for single-station operations

## Test Strategy

### Concurrency Tests (the differentiator)
1. **Last-bike race**: 50 async tasks -> 1 bike. Assert: exactly 1 success, 49 conflicts, available_bikes = 0
2. **No-oversell proof**: 200 tasks -> 10 bikes. Assert: exactly 10 success, available_bikes = 0 (not negative)
3. **Search-under-mutation**: concurrent reads (nearest search) while writes (reservations) happen. Assert: all returned `available_bikes` >= 0, no HTTP 500 errors. PostgreSQL MVCC guarantees consistent snapshots, so reads may be slightly behind, which is acceptable.
4. **Throughput**: 1000 reserves across 1000 different stations (tests system throughput, not advisory lock serialization). Target: < 2 seconds

**Note on advisory lock serialization:** Tests 1-2 intentionally serialize at the DB level for the same station. Wall time reflects correct serialization behavior, not a bottleneck. Test 4 uses different stations to measure actual system throughput. All concurrency tests hit the HTTP API via `httpx.AsyncClient`, not direct function calls, to test the full stack.

### Spatial Tests
5. **KNN correctness**: verify returned stations are actually the K nearest (compare with brute-force)
6. **Index proof**: EXPLAIN ANALYZE showing Index Scan, not Seq Scan
7. **Scale test**: 10,000 stations, verify query time stays < 10ms

### Integration Tests (CI)
8. **Full flow**: create station -> search nearby -> reserve -> return -> verify availability restored
9. **Edge cases**: reserve from empty station, delete active station, search with no results

### Stress Test Plan
The spec explicitly asks for stress test documentation showing how the system scales. This is a deliverable, not optional.

- **Load tool:** locust or raw asyncio with httpx (locust preferred for its web UI and built-in reporting)
- **Load profile:** ramp from 10 to 500 concurrent users over 5 minutes, 70% nearest-search / 30% reserve
- **Metrics collected:** p50/p95/p99 latency, error rate (%), DB connection pool usage, requests/second
- **Scaling analysis:** run the same load profile against 300 stations, 1,000 stations, 5,000 stations, 10,000 stations. Plot latency vs. station count to prove O(log n) spatial search scales.
- **Output:** formatted tables + latency distribution chart for README

## GitHub Actions CI

```yaml
name: Integration Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Start services
        run: docker compose up -d --wait
      - name: Run migrations
        run: docker compose exec api alembic upgrade head
      - name: Seed data
        run: docker compose exec api python scripts/seed.py
      - name: Run tests
        run: docker compose exec api pytest -v --tb=short
      - name: Teardown
        if: always()
        run: docker compose down -v
```

Simple, sequential, `act`-compatible. No matrix builds, no caching, no cleverness.
